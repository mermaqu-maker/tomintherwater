> 실행 파일: [supabase/migrations/0001_init.sql](../supabase/migrations/0001_init.sql)
> 스택: Supabase (Postgres 15+, Auth, Storage, RLS)

## 0. 전체 구조 한눈에

```
auth.users ─── admins (1인 소유자, is_admin() 판별)

projects ──┬── project_images     (Masonry용 width/height, 대표 1장)
           ├── project_comments   (닉네임+비밀번호 해시, 프로젝트 단위)
           └── featured_projects  (홈 Featured 순서)

products ──── reviews.product_id  (on delete set null + label 스냅샷)

costumes      (독립 카탈로그)

views: v_gallery_places / v_gallery_years   (필터 자동 생성)
rpc:   increment_project_like / add·update·delete_project_comment
storage: 'media' 버킷 (public read / admin write)
```

테이블 8개 + 뷰 2개 + RPC 4개. 이 이상 쪼개지 않은 이유: 1인 운영 사이트라 조인 비용보다 관리 단순성이 우선.

## 1. 테이블별 설계 근거

### `projects`
| 결정 | 근거 (기획 문서) |
|---|---|
| `types text[]` (editorial/event/commercial, CHECK로 값 제한) | 성격 필터는 **관리자가 등록 시 복수 지정** (2.3). enum이 아닌 배열: 한 프로젝트가 상업+이벤트 동시 가능 |
| `location text` 단독 컬럼 (credits jsonb에 묻지 않음) | PLACE 필터가 **distinct 집계로 자동 생성**되어야 하므로 인덱스 가능한 1급 컬럼으로 승격. 등록 폼에서 기존값 자동완성 → 표기 중복 방지 |
| `shot_date date` | YEAR 필터 자동 추출 소스 + 상세 "촬영 일시" 표기 |
| `like_count` 컬럼 (별도 likes 테이블 없음) | 좋아요는 "중복 방지 없이 자유 클릭" 확정 → 행 단위 기록 불필요, 카운터로 충분 |
| `published` | 관리자 임시저장/비공개. RLS 읽기 조건 |
| `collaborators`, `gear`는 text | 표준 크레딧 포맷 표시용. 검색/필터 요구 없어 정규화 불필요 |

### `project_images`
- `width/height NOT NULL` — 가로/세로 혼합 Masonry의 서버 렌더·CLS 방지에 필수 (업로드 시 추출해 저장).
- `is_cover` + **부분 유니크 인덱스** — 프로젝트당 대표 1장을 DB 레벨에서 보장 (순환 FK 없이).
- `alt` — 접근성 요구사항.

### `featured_projects`
- `projects.is_featured` 플래그 대신 별도 테이블: 홈 Featured는 **순서(position)** 가 필요하고, `−`(제외)가 프로젝트 삭제와 무관해야 함. `position unique`로 정렬 보장.

### `project_comments`
- **프로젝트 단위** (사진 단위 아님 — 라이트박스는 감상 전용 확정).
- `password_hash` = pgcrypto `crypt(pw, gen_salt('bf'))` (bcrypt). 수정/삭제 시 `crypt(입력, hash) = hash`로 재검증 → RPC 3종.
- `is_hidden` — 관리자 모더레이션은 hard delete도 가능하지만 숨김 옵션 병행.

### `products`
- `group_key` enum (essential/special/group) — Course 3그룹 드롭다운/탭과 1:1.
- `price_kind` enum (**fixed / from / inquire / coming**) — 시안에서 확정된 4가지 가격 표시 유형을 데이터로 구동. CHECK로 fixed/from일 때 `price_amount` 필수.
- `features text[]` — 불렛(비고 통합 확정)을 배열 순서 그대로 렌더.
- `example_paths text[]` — "View sample shots" 라이트박스 세트.

### `costumes`
- `type` enum 7종 = 확정 유형 (수영복/슈트/드레스/치마/머메이드테일/컨셉/소품).
- `sizes` CHECK ⊆ {44,55,66,77,FREE}, `price` CHECK ∈ {1~5만, 만원 단위} — 기획 확정값을 DB 제약으로.
- `courses text[]` — 의상 난이도별 가능 코스 필터 (긴 드레스=premium 이상). GIN 인덱스로 배열 필터.
- 이미지 메타(치수) 불필요한 카드형이라 `image_paths text[]`로 단순화 ([0]=대표).

### `reviews`
- `product_id FK on delete set null` + `product_label` **스냅샷** — 상품이 삭제/개편돼도 기존 후기의 상품 필터·표시가 유지됨.
- `rating smallint 1~5`, 부분 인덱스 `where not is_hidden` — 목록 쿼리(최신순·노출분만) 최적화.

## 2. 필터 자동 생성 (핵심 요구)

> "장소나 연도를 미리 정해놓을 수 없다" → **등록된 데이터에서 자동 파생**

- `v_gallery_places`: published 프로젝트의 `location` distinct + 카운트 (카테고리별)
- `v_gallery_years`: `shot_date`의 연도 distinct + 카운트
- 프런트는 이 뷰만 조회하면 PLACE ▾ / YEAR ▾ 드롭다운이 자동 구성. 새 프로젝트 등록 즉시 반영.
- `security_invoker = true` → 뷰도 RLS를 통과하므로 비공개 프로젝트는 집계에 안 잡힘.

## 3. 쓰기 경로와 보안 (RLS 전략)

```
방문자(익명)                            관리자(1인)
  │ 읽기: published/노출분만 (RLS)        │ Supabase Auth 로그인
  │ 좋아요: increment_project_like RPC    │ admins 테이블 등재 → is_admin()
  │        (anon 직접 호출 허용)          │ RLS로 전 테이블 CRUD + Storage 쓰기
  │ 후기·댓글 작성/수정/삭제:
  └─▶ Next.js Server Action
        ├ Turnstile 캡차 검증 + rate limit
        └ service_role로 RPC/insert 호출   ← anon에겐 RPC execute 자체를 미부여
                                             → API 직접 호출로 캡차 우회 불가
```

- **admins 테이블 + `is_admin()`**: "authenticated면 관리자"보다 안전(향후 계정 추가 대비). 배포 시 소유자 uid 1행 insert.
- **댓글 비밀번호**: 평문 저장 금지 — bcrypt 해시, 검증은 DB 함수 내부에서만. 클라이언트에 해시도 노출 안 됨(RPC가 boolean만 반환).
- **좋아요만 anon RPC 직접 허용**: "마구 눌러도 됨" 확정 사양이므로. 남용은 엣지 rate limit으로 완화.
- **Storage `media` 버킷**: public read(CDN), 쓰기는 관리자만. 후기 첨부 이미지는 서버 경유 업로드(service_role).

## 4. UI ↔ 스키마 매핑 (검증표)

| 화면/기능 (시안) | 스키마 |
|---|---|
| 갤러리 수중/지상 탭 | `projects.category` |
| ALL/EDITORIAL/EVENT/COMMERCIAL | `projects.types` GIN |
| PLACE ▾ / YEAR ▾ | `v_gallery_places` / `v_gallery_years` |
| Masonry (원본비율·CLS) | `project_images.width/height` |
| 상세 크레딧 (장소/협업자/장비/일시) | `location, collaborators, gear, shot_date` |
| 프로젝트 좋아요 | `like_count` + RPC |
| 댓글 작성/비번 수정·삭제/관리자 삭제 | `project_comments` + RPC 3종 + RLS |
| 홈 Featured −/＋/순서 | `featured_projects.position` |
| Course 3그룹 + 가격 4유형 | `products.group_key, price_kind` |
| View sample shots | `products.example_paths` |
| 의상 필터 (유형/사이즈/컬러/코스) | `costumes` 배열 컬럼 + GIN |
| 후기 상품 필터 + 별점 | `reviews.product_label, rating` |
| 관리자 인라인 편집 전체 | RLS `is_admin()` write 정책 |

## 5. 적용 순서 (배포 시)

1. Supabase 프로젝트 생성 → `supabase db push` (또는 SQL Editor에서 0001 실행)
2. Auth에서 소유자 계정 생성 → `insert into admins (user_id) values ('<uid>')`
3. Auth 설정: 재설정 리다이렉트 URL `/admin/reset` 허용 목록 등록
4. 시드: 기존 사이트 프로젝트 데이터 + Course 5종 입력 (시드 스크립트는 M2에서 작성)
